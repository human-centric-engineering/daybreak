### TL;DR

**What this is.** The design for the three missing concepts of the five-concept expert-led-app framework — **Modules**, **Facilitation Structures**, and **Data-Slots** — as new domains inside Sunrise. Knowledge Base and Agents already exist; these three compose them into guided, personalised experiences. The framework is domain-agnostic: every app, whatever its domain, is configuration and content — never framework code.

**The core stance.** Nothing here invents new kinds of machinery. Sunrise already contains every pattern the framework needs — immutable version chains, a capability registry, guards/supervisor/evaluations, an approval queue, insert-only event history, a graph canvas — and this design instantiates those patterns one level up the stack. Two decisions organise everything:

- **Definition vs state, everywhere.** Authored definitions (module config, the journey map, slot definitions, policies) get immutable version chains; per-user data (slot values, journey events) gets insert-only history. Everything is timestamped; nothing is ever destructively overwritten.
- **Code-first modules.** A module is a bounded code unit registered through a seam (like capabilities); the database holds only what operators control — status, flags, availability windows, and parameters whose admin form renders generically from each module's declared schema.

**The key points, one per concept.**

- **Modules** are features with universal controls (lifecycle, feature-flag binding, time windows) plus per-module parameters; agents, workflows, and knowledge attach by _binding_, never ownership, so universal and module-specific agents fall out of one mechanism. Stats derive from an insert-only event stream, never counters.
- **Facilitation Structures** decompose into five parts: a versioned **map** (typed graph: prerequisite / unlocks / tangent / related-to edges, plus declarative state, slot, and temporal gate conditions), per-user **journey state** (event log as source of truth), a deterministic **engine** that is the _sole writer_ of state and rejects invalid moves with reasons, an advisory **guidance** layer that ranks already-eligible options using the freshest slots, and **governance** that reuses guards, supervision, evaluations, and approvals. Self-proposed structure changes (emergence) pass one gated pipeline — validation → invariant check → approval → new version — so the deterministic spine is never written raw.
- **Data-Slots** are what the system learns about the user through conversation: insert-only versioned values carrying confidence, source type, and per-version provenance linking back to the originating conversation. Capture runs silently through ordinary capabilities; sensitivity classification, typed values, and selective per-agent exposure are built in from day one.

**Placement and status.** The framework ships in its own **fork/repo of Sunrise** (not in core); each app forks the framework repo, which tracks Sunrise via the `building-on-sunrise` model. Clean framework↔Sunrise seams are an enforced boundary from Phase 1 (Appendix B); licensing of the framework repo (open vs proprietary) is open and does not block the build. Conversations are surface-scoped; continuity travels as state, not threads. Everything in the document is Binding except three open items that genuinely cannot be decided yet (§9). Appendix A lists every decision with rationale; §10 sequences the build in six phases.

## 1. Framing

Sunrise today is an orchestration platform: agents, capabilities, knowledge bases, workflows, providers, evaluation, governance primitives. The five-concept framework adds a layer _above_ orchestration that turns those primitives into a coherent, guided user experience. Two of the five concepts (Knowledge Base, Agents) already exist as platform domains. This document designs the remaining three — **Modules**, **Facilitation Structures**, and **Data-Slots** — as new platform domains that compose the existing ones rather than duplicating them.

The single most important design stance, stated up front: **almost everything these three layers need already exists in Sunrise as a pattern, and the job is to instantiate those patterns at a new level of the stack, not to invent parallel machinery.** Versioned authored definitions exist (workflow version chains). Code-defined units registered through a seam exist (capabilities). Per-entity agent configuration exists (agent profiles, capability pivots). Governance exists (guards, supervisor, evaluation metrics, approval queue, audit log). Insert-only user history exists (events, audit). The graph canvas exists (the custom workflow-builder canvas). The framework layer should feel, to anyone who knows the orchestration layer, like more of the same discipline applied one level up.

## 3. The layer model

The framework sits between the orchestration platform and the app's own code. Reading bottom-up:

```
┌─────────────────────────────────────────────────────────────────┐
│  APP (a fork: any expert-led app)                                │
│  Concrete modules (code), slot definitions, the authored map,    │
│  agent configs, knowledge content — all configuration & content  │
├─────────────────────────────────────────────────────────────────┤
│  FRAMEWORK LAYER (this design)                                   │
│                                                                  │
│  ┌────────────┐   reads state,     ┌──────────────────────────┐  │
│  │  MODULES    │◄──gates access────│  FACILITATION STRUCTURES  │  │
│  │  registry + │                   │  map (authored, versioned)│  │
│  │  config +   │──emits events────►│  + journey state (runtime)│  │
│  │  bindings   │                   │  + deterministic engine   │  │
│  └─────┬──────┘                    │  + governance policies    │  │
│        │ writes/reads              └───────────┬──────────────┘  │
│        ▼                                       │ reads freshest   │
│  ┌──────────────────────────────────────────── ▼ ─────────────┐  │
│  │  DATA-SLOTS — definitions (authored) + values (insert-only │  │
│  │  versioned user data, confidence, source, provenance)      │  │
│  └─────────────────────────────────────────────────────────────┘ │
├─────────────────────────────────────────────────────────────────┤
│  ORCHESTRATION (exists): agents, capabilities, knowledge,        │
│  workflows, guards, supervisor, evaluations, approvals, hooks    │
├─────────────────────────────────────────────────────────────────┤
│  PLATFORM (exists): auth, privacy/erasure, flags, audit,         │
│  analytics, rate limits, admin shell                             │
└─────────────────────────────────────────────────────────────────┘
```

Two structural decisions follow from this picture and govern everything below.

**Decision A — definition vs. state, everywhere.** Each of the three domains splits into an _authored definition_ side (versioned with immutable chains, edited by admins/experts/engineers, eventually proposable by agents) and a _runtime state_ side (per-user, insert-only history, GDPR-erasable). This is the same split the workflow domain already makes (`AiWorkflow` definition vs `AiWorkflowExecution` runtime), applied uniformly. It gives one consistent answer to the "everything versioned and timestamped" principle, stated once here and repeated verbatim wherever it applies: **authored definitions get immutable version chains; user data gets insert-only history; everything is timestamped; nothing is ever destructively overwritten.**

**Decision B — code-first units, config-second.** A module is a bounded area of the codebase, so the framework treats modules the way it treats capabilities: the _code_ defines what a module is and can do (registered through a seam at boot), and the _database_ holds only the per-deployment configuration, lifecycle, bindings, and stats for that registered unit. The DB never tries to describe code that doesn't exist; the code never hard-codes values an admin should control. The same code/config split is echoed in data-slots (definition rows are seedable config, capture logic is code) and facilitation (the engine is code, the map is data).

### 3.1 Proposed placement in the repo

```
prisma/schema/
  framework-modules.prisma          # module config, bindings, stats
  framework-facilitation.prisma     # graph, journey state, policies, proposals
  framework-data-slots.prisma       # slot definitions, values, provenance

lib/framework/
  modules/        # registry, lifecycle service, stats service, seam
  facilitation/   # graph store (interface + Postgres impl), engine,

  data-slots/     # capture service, capabilities, provenance, erasure
  shared/         # scope types, version-chain helpers shared by the three

app/admin/framework/
  modules/        # list, config form, stats
  facilitation/   # map editor (workflow-builder canvas), journey explorer, policies
  data-slots/     # definition manager, per-user inspector (support tooling)

app/api/v1/admin/framework/...     # admin APIs (API-first, per CLAUDE.md)
app/api/v1/framework/...           # consumer-facing runtime APIs (later)
```

A single `lib/framework/` umbrella (rather than three top-level domains) mirrors how `lib/orchestration/` holds many interlocking subdomains, and keeps the framework's internal coupling — facilitation reading slots, modules emitting journey events — inside one boundary. Schema files use a `framework-` prefix so the domain split in `prisma/schema/` stays legible alongside the `orchestration-` family. The framework lives in its own fork/repo of Sunrise (not in core), structured so its only connections to Sunrise run through registration seams, enforced by a boundary rule from Phase 1; the full rationale is in **Appendix B**. Each in-genre app forks the framework repo, carrying concrete modules, maps, and slot definitions: the registration seams (`registerModule()` etc.) join Sunrise's public-surface contract in `VERSIONING.md`, and `lib/app/modules.ts` becomes a fork-owned scaffold like `lib/app/capabilities.ts`.

## 4. Modules

### 4.1 What a module is

A module is a **registered, bounded feature unit**: a directory of code with a declared identity, plus a database row that gives admins control over it. The code half declares everything intrinsic to the module — what it's called, what configuration it accepts, which capabilities it brings, what events it emits. The database half holds everything an operator decides — whether it's on, who sees it, what its parameters are set to, when it's available.

**Code half — the `ModuleDefinition`.** Each module exports a definition object and registers it at boot through a fork-owned seam, exactly like capabilities:

```typescript
// lib/app/modules.ts  (fork-owned scaffold, ships empty in Sunrise)
export function initAppModules(): void {
  registerModule(coreValuesModule); // app's own modules
  registerModule(checkInsModule);
}

// A module's own code, e.g. modules/<slug>/definition.ts in the fork
export const someModule: ModuleDefinition = {
  slug: 'some-module', // stable identity, referenced everywhere
  name: 'Some Module',
  description: '…',
  configSchema: z.object({
    // module-specific admin parameters
    sessionLengthTarget: z.number().min(1).default(3),
    tone: z.enum(['gentle', 'direct']).default('gentle'),
    // …whatever this module's functionality needs
  }),
  capabilities: [/* BaseCapability instances scoped to this module */],
  slotDefinitions: [/* slot definitions this module owns — see §6 */],
  events: ['entered', 'progressed', 'completed', 'abandoned', 'feedback_given'] as const,
  agentRoles: ['companion', 'specialist'] as const, // named seats agents can fill
};
```

The Zod `configSchema` is the key move: **module-specific parameters vary per module, but the admin UI doesn't need to know any of them in advance.** The admin module-config form renders generically from the schema (Sunrise already has the form-builder discipline and `<FieldHelp>` convention for this), validates with the same schema on the API, and stores the values as JSON on the module's DB row. New module, new parameters, zero admin-UI work.

**Database half — the `Module` row and its universal parameters.** Registration upserts a row keyed by slug (the same sync-on-boot pattern the capability registry uses). The row carries the _universal_ per-module controls — the parameters every module has regardless of functionality:

```prisma
// framework-modules.prisma  (sketch)
model Module {
  id              String    @id @default(cuid())
  slug            String    @unique          // matches the registered ModuleDefinition
  name            String                     // display override; defaults from code
  status          String    @default("draft")// draft | active | scheduled | retired
  featureFlagName String?                    // optional binding to FeatureFlag — one switch
  availableFrom   DateTime?                  // limited-time modules: window start
  availableUntil  DateTime?                  // …window end; both null = long-lived
  audience        String    @default("all")  // all | invite | flag-gated (room to grow)
  config          Json      @default("{}")   // values validated by the module's configSchema
  configHistory   Json      @default("[]")   // or a ModuleVersion table — see Decision 2
  isRegistered    Boolean   @default(true)   // false = code removed but row retained (audit)
  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt

  agentBindings    ModuleAgentBinding[]
  workflowBindings ModuleWorkflowBinding[]

  @@index([status])
  @@map("framework_module")
}
```

_Effective availability_ is computed, never stored: a module is live for a user iff `status = active` **and** its feature flag (if bound) is enabled **and** now is inside the availability window **and** — crucially — the facilitation layer says this user has reached it (§5). The module system answers "is this module on, at all?"; facilitation answers "is it open _to this user, now_?". Keeping those two questions in separate layers is what keeps limited-time modules, flag rollouts, and journey gating from tangling.

Module config is versioned with a full **`ModuleVersion` snapshot table** (the `AiAgentVersion` pattern), not an append-only JSON history column: it honours the "everything versioned" principle with the same mechanism admins already know from agents and workflows, it makes "what changed and who changed it" first-class, and experts will eventually edit module config, so rollback matters. The accepted cost is one more table and one more form tab per module — and the `configHistory Json` field in the sketch above is replaced by a `ModuleVersion` relation at implementation time.

### 4.2 Agents, capabilities, workflows — bindings, not ownership

The requirement — some agents only make sense inside a module, others are universal — maps onto a **binding** model rather than agents "belonging" to modules. Agents stay ordinary `AiAgent` rows managed in the existing admin pages. A pivot binds them into a module with a _role_:

```prisma
model ModuleAgentBinding {
  id        String  @id @default(cuid())
  moduleId  String
  agentId   String
  role      String                    // one of the module's declared agentRoles
  isPrimary Boolean @default(false)   // e.g. the chat-companion seat
  config    Json?                     // per-binding overrides (tone hints, etc.)

  module Module  @relation(fields: [moduleId], references: [id], onDelete: Cascade)
  agent  AiAgent @relation(fields: [agentId], references: [id], onDelete: Cascade)

  @@unique([moduleId, agentId, role])
  @@map("framework_module_agent")
}
```

A universal agent is simply one bound into many modules (or invoked outside any module scope); a module-specific agent is bound into one. Nothing on `AiAgent` changes. The module's declared `agentRoles` act like named seats — the admin binds an agent into each seat, and the same generic admin UI works for every module.

**Capabilities** scope the same way they already do — through the existing `AiAgentCapability` pivot on whichever agents are bound to the module. Module-declared capabilities are registered into the global registry at boot, namespaced by module slug (e.g. `some-module.save_worksheet`) — there is one flat capability registry platform-wide, so the namespace prefix is what prevents two modules from colliding on a generic tool name — and then granted to the right agents. No second capability system. The one addition worth making is deliberately _generic_: the dispatcher's `CapabilityContext` gains an optional `scope?: Record<string, string>` extension map — core code never names a framework concept — and the framework populates `scope.moduleSlug` when dispatching in module context, so a capability can know which module scope it's executing in (and refuse outside it). The genericity is a boundary requirement, not a style choice: upstream Sunrise must contain zero framework vocabulary, so the seam survives every re-base (Appendix B).

**Workflows** bind via the existing trigger machinery. Module lifecycle events (`entered`, `completed`, `feedback_given`, plus module-specific ones) are dispatched through the existing event-hook system, and a thin `ModuleWorkflowBinding` (moduleId, eventType, workflowId, enabled) lets admins say "when X happens in this module, run workflow Y" — reusing `AiWorkflow` and its execution engine untouched.

**Knowledge** needs no new mechanism at all: the agents bound to a module are given document/tag grants through the existing restricted-access system, which is precisely "this module's agents only see this module's corner of the expert's material."

### 4.3 Stats and engagement

Module stats are **derived from an event stream, never maintained as counters.** Rationale: counters drift (every missed or double-counted increment is permanent and undetectable), cannot be recomputed when a bug is found, and can only answer the questions chosen before the data arrived — whereas an insert-only event stream computes any future metric retroactively over the full history. The framework writes one insert-only table of engagement events (which doubles as the facilitation layer's journey log — see §5.4, they are the same stream):

```
JourneyEvent { userId, moduleSlug?, nodeKey?, type, payload Json, occurredAt }
   types: module.entered, module.progressed, module.completed,
          module.feedback (payload: rating/comment), session.started, …
```

The admin module page then shows engagement (unique users, entries, completion rate, dwell, return rate) and feedback (ratings distribution, recent comments) computed from this stream — the same pattern as the existing orchestration analytics (popular topics, engagement) which already aggregates from conversations. User-facing feedback capture itself arrives through a built-in framework capability (`record_feedback`) any bound agent can call, plus a plain API endpoint for UI-driven feedback.

### 4.4 What "bounded area in the codebase" means concretely

Each module in a fork is a directory (e.g. `modules/<slug>/` at the fork level) containing its definition, its capabilities, its slot definitions, its server logic, and later its UI components and API route handlers. Two enforcement tools keep the boundary real, both already in the repo's toolbox: an **ESLint boundary rule** (the pattern used for `lib/app/**`) preventing cross-module imports except through declared public exports, and the **module registry** as the only path by which the platform learns a module exists. Backend-only for now per the stated scope — but the directory layout should reserve the `ui/` and `api/` subfolders so the later front-end work lands inside the same boundary.

## 5. Facilitation Structures

This is the most novel of the three layers, and the one with the widest brief: routes and navigation, moment-to-moment guidance, governance and supervision, adaptivity and (eventually) emergence. The way to keep that breadth from collapsing into mush is to be very strict about its internal anatomy. The layer decomposes into five distinct parts, each with a different nature, each reusing a different existing Sunrise pattern:

| Part                 | Nature                                                         | Reuses                                                          |
| -------------------- | -------------------------------------------------------------- | --------------------------------------------------------------- |
| **1. The Map**       | Authored, versioned definition (a typed graph)                 | Workflow version-chain discipline; workflow-builder canvas      |
| **2. Journey State** | Per-user runtime state + insert-only event log                 | Satellite-table + erasure conventions; execution/event patterns |
| **3. The Engine**    | Deterministic, pure code — no LLM                              | Workflow DAG validator style; platform-agnostic lib code        |
| **4. Guidance**      | Advisory layer: ranks options using fresh slots (+ similarity) | Capability registry; pgvector; context builder                  |
| **5. Governance**    | Policies + supervision + change-control                        | Guards, supervisor, evaluations, approval queue, audit log      |

The deterministic engine is the spine. Everything probabilistic (LLM narration, slot-informed ranking, similarity) reads _from_ it and never writes _to_ it except through validated transitions. That single rule is what makes the layer safe to make adaptive later: emergence gets to propose, the spine gets to decide.

### 5.1 The Map — authored structure

Model the map as a **typed property graph stored in Postgres behind a `GraphStore` interface**. Nodes are places and structural markers; edges are typed relationships:

```prisma
// framework-facilitation.prisma (sketch)
model FacilitationGraph {            // one row per named map; usually one per app
  id                 String  @id @default(cuid())
  slug               String  @unique
  name               String
  draftDefinition    Json?            // in-progress edit (draft-on-edit)
  publishedVersionId String?          // current live version
  versions           FacilitationGraphVersion[]
  @@map("framework_facilitation_graph")
}

model FacilitationGraphVersion {     // immutable snapshot — mirrors AiWorkflowVersion
  id           String   @id @default(cuid())
  graphId      String
  version      Int
  definition   Json     // { nodes: [...], edges: [...] } — full snapshot
  changeSummary String? @db.Text
  createdBy    String?  // user id, or "agent:<slug>" for approved proposals
  createdAt    DateTime @default(now())
  @@unique([graphId, version])
  @@map("framework_facilitation_graph_version")
}
```

Inside `definition`, nodes and edges are plain JSON with stable keys:

```
node: { key, type: module | stage | milestone | region,
        moduleSlug?,            // type=module → binds to a registered Module
        stage?,                 // maturity level this node belongs to
        region?,                // key of the containing region node (see below)
        completionMode?,        // once (default) | repeatable — recurring places
                                // (check-ins, practice sessions) never "complete shut"
        onFirstArrival?: {...}, // "airport" semantics: one-time welcome workflow/agent
        meta }
edge: { from, to, type: prerequisite | unlocks | tangent | related_to,
        condition?: {...},      // optional richer gate (see below)
        meta }
```

**Regions are first-class containers in the format, from day one.** A small map may be thirty nodes; a curriculum-scale map runs to hundreds. Whole-map snapshot versioning remains the right consistency model regardless of size, but the JSON format must support hierarchy — region nodes that _contain_ other nodes — so the editor can collapse/expand at region level, diffs and validation can report at region granularity, and a future "import a structured curriculum → draft map" path (which is just the §5.5 proposal pipeline with a different author) has containers to target. This is purely a format decision now, at zero implementation cost; retrofitting hierarchy into a flat format after large maps exist is the expensive version.

Four format decisions follow, each binding, each stated with its rationale.

**Map decision 1 — stable string `key`s, not row ids.** User journey state references node keys. Rationale: a republished map must not orphan anyone's progress — the same reason workflow executions pin to a version while the workflow keeps its slug. Runtime state is _interpreted against the currently published version_ (state stores where the user has been; the live map decides what that means now), with a validation pass at publish time warning the admin when a key with live user state is being removed or re-gated.

**Map decision 2 — the whole map versions as one snapshot**, never nodes/edges row-by-row. Rationale: maps are authored and reviewed as a whole, exactly like workflow definitions, and snapshot-versioning is the discipline the codebase already trusts (`AiWorkflowVersion`); row-level versioning would invent a second, harder consistency model for no authoring benefit.

**Map decision 3 — typed edges over generic links.** Exactly four types: `prerequisite` (all incoming must be satisfied), `unlocks` (any one suffices), `tangent` (always open from here, no completion semantics), `related_to` (advisory only, feeds guidance). Rationale: these four cover the map-of-towns semantics — gated cities, open-world regions, side-trips — without a rules language, and a closed set is what keeps the engine's availability computation total and explainable.

**Map decision 4 — the `condition` format is family-tagged and forward-compatible; three predicate families are defined now.** The optional `condition` field on edges is the escape hatch for richer gates, expressed as declarative JSON the engine evaluates — never as code in the map. Every condition carries an explicit family tag (`{ family: "state" | "slot" | "temporal", … }`), and publish-time validation rejects _unknown_ families rather than the format forbidding them — so a future family is an additive engine capability plus a validator update, never a migration of published, versioned maps. (Without the tag, adding a family later would change the persisted map format, which snapshot versioning makes expensive; the tag is what keeps that door cheap.) The families defined now: **state predicates** ("milestone M reached"), **slot predicates** ("slot X at confidence ≥ 7", "slot Y ≥ value" — the reason slot values gain a typed form in §6.1), and **temporal predicates** (`available_after`, `available_until`, `recommended_by`, `cooldown_since_last_visit`). Temporal predicates make the engine _time-aware_ — evaluated against the clock and journey timestamps — which is what deadline-driven apps (hard dates, spaced revisiting) and seasonal content need, with no scheduler and no new machinery: the engine takes `now` as an input like any other.

**Storage: Postgres now, interface forever.** The concept notes deliberately leave the storage engine open (Postgres + recursive queries vs a dedicated graph database); this design commits to the _interface_, not the engine. `GraphStore` exposes exactly the operations the framework needs (`getPublishedGraph`, `validate`, `reachableFrom`, `availabilityFor(state)`, `neighbours`, `pathsBetween`), implemented first with Prisma + recursive CTEs. An authored map is small and bounded (tens to low hundreds of nodes), which recursive SQL handles comfortably; a dedicated graph database earns its operational cost only if collective-journey analytics later demand deep pathfinding at scale — and the interface means adopting one is an implementation swap, not a rewrite. This honours the "build it properly, choose the engine deliberately" intent without taking on a second datastore on day one.

The **pgvector overlay** rides on the same nodes: each node (and optionally each slot-group) gets an embedding derived from its description/content, stored in a small `framework_node_embedding` table using the same `Unsupported("vector(1536)")` + HNSW pattern as knowledge chunks. The contract from the concept notes is preserved structurally: _authored edges are the only thing the engine reads for eligibility; similarity is only ever consulted by the guidance layer for ranking and "what else might resonate."_ Two different parts of the anatomy, so the boundary can't blur.

### 5.2 Journey State — runtime traversal

Per-user state follows the satellite convention exactly (plain `String` userId FK, hand-written `ON DELETE CASCADE`, erasure-hook registration):

```prisma
model UserJourney {                  // one per user per graph PER CONTEXT
  id        String  @id @default(cuid())
  userId    String
  graphSlug String
  contextKey String @default("")    // distinguishes parallel instances of the same
                                    // map for one user (e.g. one user supporting two
                                    // different people through the same map — see §8).
                                    // "" = the default, context-free journey.
                                    // NOT NULLABLE, by design: Postgres treats NULLs
                                    // as distinct in unique indexes, so a nullable
                                    // contextKey would let duplicate (userId, graphSlug)
                                    // default journeys slip past the @@unique below.
                                    // The empty-string sentinel is load-bearing.
  startedAt DateTime @default(now())
  nodeStates UserNodeState[]
  @@unique([userId, graphSlug, contextKey])
  @@map("framework_user_journey")
}

model UserNodeState {                // current standing per node — fast reads
  id          String   @id @default(cuid())
  journeyId   String
  nodeKey     String
  status      String   // unvisited | available | active | visited | completed
  timesCompleted Int   @default(0)  // repeatable nodes (completionMode: repeatable)
                                    // count completions; status reflects the latest pass
  progress    Json?                 // optional module-defined progress payload
                                    // (e.g. % through a unit, mastery levels) — shape is
                                    // owned by the node's module; the engine treats it
                                    // as opaque. Deeper attainment data belongs in slots.
  firstEnteredAt DateTime?           // powers "firsts" / airport semantics
  lastActiveAt   DateTime?
  completedAt    DateTime?           // most recent completion for repeatable nodes
  @@unique([journeyId, nodeKey])

}

  journeyId  String

  occurredAt DateTime @default(now())
  @@index([journeyId, occurredAt])

}
```

`UserNodeState` is technically a projection of the event log (current status is derivable by replay), kept materialised for cheap reads — the same projection-over-log relationship `AiWorkflowExecution` has to its step events. The event log is the source of truth, the synopsis agents' raw material, the stats feed, and the admin's individual-journey timeline, all at once.

### 5.3 The Engine — deterministic spine

A pure, platform-agnostic library (`lib/framework/facilitation/engine/`) with two jobs and no LLM anywhere near it:

**Availability computation.** `computeAvailability(publishedGraph, nodeStates, moduleLiveness, now) → { perNode: available|locked|…, validMoves, firsts }`. It evaluates typed edges and conditions (including the temporal predicates — `now` is an explicit input, which also keeps the function pure and trivially testable) against state, intersects with module-level liveness (§4.1 — a journey-unlocked node whose module is flag-disabled is still closed), and returns the complete, explainable picture. _Explainable_ matters: every locked node carries its reason ("requires completing X", "opens after date D"), because the guidance layer and the user-facing narration must never have to guess.

**Transition validation.** `applyEvent(state, event) → newState | rejection`. Entering a node requires it to be available; completions follow the node's declared semantics — a `once` node closes, a `repeatable` node increments `timesCompleted` and reopens (subject to any `cooldown_since_last_visit` condition), which is how recurring places (check-ins, practice sessions) live in the same state machine as one-way milestones; every accepted event appends to `JourneyEvent` and updates the projection in one transaction. Agents and APIs _request_ transitions; the engine is the only writer. This is the property that lets agents safely operate the rails: a hallucinated `enter_module("locked-thing")` is rejected with a structured reason, not obeyed.

The engine also hosts **invariant validation** for the authoring side — publish-time checks that the map has no cycles among prerequisites, no unreachable required nodes, no edge that violates a declared gating rule. This mirrors the workflow DAG validator, and it's the same machinery the self-generation path (§5.5) runs proposals through.

### 5.4 Guidance — "what would serve the user best right now"

This is the advisory layer that reads the engine's output plus the **freshest data-slots** and produces ranked, reasoned suggestions: _linger here_ vs _move on_, which valid next step to surface first, whether to suggest a tangent. Architecturally it is a service (`guidance.ts`) plus a family of **built-in framework capabilities** through which agents — and only agents with these capabilities granted — consume it:

```
get_journey_state()        → engine state, beautifully structured (state/orientation agents)
get_next_steps()           → valid moves + guidance ranking + reasons (path agents)
get_progress_synopsis()    → event-log digest scoped for narration (synopsis agents)
suggest_focus()            → linger/move recommendation + reasoning (facilitator voice)
request_transition(nodeKey)→ asks the engine to move; may be user-confirmed first
```

The ranking inside `get_next_steps`/`suggest_focus` is where fresh slots matter: recency-weighted slot reads (low-confidence areas, recently changed values, declared preferences) score the engine's _already-eligible_ options, and the pgvector overlay contributes "related places" — advisory, clearly labelled as such in the payload. The division of labour required by the concept notes is enforced by construction: **the engine computes what is possible; guidance ranks what is wise; agents narrate and converse — reading from the engine, never guessing.**

**Proactive guidance is a named responsibility of this layer, not an afterthought.** Reactive guidance answers when the user is present; deadline-driven and pacing-driven apps also need the system to reach _out_ — "your deadline is in six weeks and region X is untouched", "it's been a while since your last check-in". Architecturally this costs nothing new: a scheduled workflow (the existing `AiWorkflowSchedule`) periodically runs the same guidance evaluation over active journeys and, where a nudge is warranted, dispatches it through the existing event-hook/notification machinery. The point of naming it here is to keep one rule intact: outbound nudges are _guidance outputs_ — produced by the same slot-and-engine-informed ranking, subject to the same governance — never a separate bolt-on messaging system with its own opinions.

The **facilitation agent family** (onboarding/discovery, orientation, synopsis, state, path/progress — and the distinct facilitator/supervisor persona the user occasionally meets) needs no new agent machinery: they are `AiAgent` rows bound to the facilitation layer through the same binding pattern as modules (a `FacilitationAgentBinding` with `role`), granted the capabilities above instead of module tools. One binding mechanism, two scopes.

**Where this meets a live conversation:** the chat context builder. When a conversation runs in module scope, the context assembled for the agent includes the module's config-relevant context, the user's journey position, and selected fresh slots — kept current per turn. The mechanism is a generic **context-contributor registry** on `buildContext()`: core exposes a registration point for context contributors; the framework registers its contributor through `initFramework()` like any other consumer; core never imports framework code or names a framework concept. This is the single integration point the framework needs inside the existing chat pipeline, and keeping it registry-shaped is a boundary requirement (Appendix B): Sunrise's `buildContext()` stays framework-agnostic, so upstream re-bases never touch it.

**Conversation scoping: conversations are surface-scoped; continuity is carried by state, not by threads.** Each module's chat companion is its own `AiConversation` with the module's bound primary agent — directly following the framework overview, in which each module "usually has its own Chat Companion Window". Facilitation surfaces (welcome, check-ins, orientation, synopsis moments) are likewise their own conversations with facilitation-bound agents. Moving between modules is a journey transition, not an agent handoff inside one thread: the user leaves one conversation surface and arrives at another, and what travels with them is the framework's state — slots, journey position, progress synopsis — injected fresh via `buildContext()` on the new surface. Rationale: this matches the source documents' model; it avoids mid-thread agent identity swaps (confusing for users, and hard to govern — guard modes, knowledge grants, and capability sets are all per-agent); and it reuses the existing conversation model entirely unchanged. The "one companion everywhere" pattern some apps may want remains expressible without any framework change: it is a facilitation-scoped agent whose surface persists across the app — an app-level configuration choice, not a routing problem.

### 5.5 Governance — two faces, one layer (and the path to emergence)

The concept notes flag an open tension: facilitation (gentle, user-felt) and governance (rules, supervision) as two faces of one layer, with a considered-and-rejected split into separate concepts. The architecture supports keeping them as one layer precisely _because_ the anatomy above already separates concerns internally — governance is simply parts 3 and 5 wearing a stricter face. Concretely, governance decomposes into three reuses and one new piece:

**Inline supervision** is the existing guard pipeline (input/output/citation guards with `log_only | warn_and_continue | block` per agent) applied to framework-bound agents — vetting a live turn before the user sees it. Nothing new to build; facilitation just becomes opinionated about guard _settings_ for agents bound into sensitive scopes. **Post-hoc supervision** is the existing supervisor assessment and evaluation metrics run over framework conversations, feeding the improvement loop. **Change control** is the existing audit log (every authored change to maps/modules/slot definitions) plus the approval queue.

The new piece is **policy** — and to stop it becoming a junk drawer, `FacilitationPolicy` is designed as several small _typed policy kinds_ under one table (a `kind` discriminator with a Zod-validated payload per kind), not one generic rules blob:

- **Relevance/maturity gating** — stage/region → allowed agent roles, optional persona switches, evaluated at conversation-routing time. The mechanism behind "which agent groupings matter most depends on how far the user has matured" — data an admin edits, not logic an agent improvises.
- **Guard minimums per scope** — designated scopes can _mandate_ guard modes (e.g. inline `block` on output guards, non-negotiable). This matters because the inline-vs-post-hoc supervision question stops being a latency/cost trade-off in safety-critical deployments (a clinical-adjacent app is the obvious case): for those scopes, inline vetting before the user sees a turn must be mandatable by policy, and the latency cost is simply accepted.
- **Escalation pathways** — a first-class policy kind, not an improvisation: _when signal S is detected in scope X, do Y_. Detection comes from the existing guards and supervisor (distress, risk, boundary signals); the response is declarative — route the conversation to a differently-configured agent, surface human contact points and resources, notify a human reviewer, and always log to the journey event stream. The machinery is entirely existing (guards detect, workflows execute, hooks notify, events record); the policy kind is what turns "the output guard blocked something" into a _defined, configured, auditable pathway_ — which a safety-critical app configures on day one rather than discovers it needs in an incident.
- **Auto-approval risk classes** — which structure-change proposals may bypass human sign-off (see below).

**Emergence, safely.** Self-configuration enters through one narrow gate: a `StructureChangeProposal` table. A proposing agent (or an admin tool, or eventually the expert-onboarding agent from the long-term aspiration) writes a proposal — a diff against the current published map, module config, or policy set. Every proposal flows the same pipeline regardless of author: **schema validation → engine invariant check → risk classification → approval → publish as a new version**, with `createdBy = "agent:<slug>"` preserved in the version history. The approval step reuses the existing approval queue; the risk classifier is a policy knob (`autoApprove: none | low_risk`) so the trade-off you named — full human sign-off vs auto-approving invariant-safe low-risk changes — is an admin setting, not an architectural fork. The deterministic spine is never written raw; emergence is real but always laundered through validation and versioning.

### 5.6 Admin experience

The map editor is the workflow builder's graph canvas re-instantiated (Sunrise's own custom canvas, not the React Flow library) with facilitation node/edge types: a palette of node types, typed edges drawn between them, region containers that collapse/expand (essential once maps reach curriculum scale), a config panel per node (gating conditions, first-arrival behaviour, module binding), validation errors surfaced before publish, and the version history / publish / rollback controls admins already know from workflows. Route _visualisation_ for operations is the same canvas in read-only mode with overlays: collective heat (visit/completion counts per node from the event stream), drop-off edges, and an individual-journey mode that replays one user's event log along the map. Journey analytics (popular paths, beneficial vs abandoned regions, cohort comparisons) aggregate from `JourneyEvent` — the existing analytics pages provide the visual language. The explorer's individual-journey view is built as one instance of a _subject scope_ (a single user today, a cohort tomorrow), so the future group view (§8) is a scope change, not a second explorer.

**The framework atlas: one zoomable picture of the whole configuration.** The map editor and journey explorer show the _geography_; a third surface shows the _composition_ — what the framework is actually made of, visually, on the same workflow-builder canvas. The atlas is read-only and works by **semantic zoom**:

- **Zoomed out** it shows the published facilitation map — regions, places, typed edges — exactly as the map editor renders it, plus the facilitation layer itself as a node (since it has its own agent family and slot scopes).
- **Zoom into any module** and its composition unfolds as satellite nodes around it: the agents bound into each declared role (with primary-seat and guard-mode badges), the slot definitions it owns (grouped, with sensitivity and visibility markers), its workflow bindings (event → workflow), its registered capabilities, and its knowledge grants. The same unfolding works on the facilitation node — its agent roles, its policies, its framework capabilities.
- **Cross-cutting lenses** invert the view: select an agent and see every module and facilitation role it is bound into; select a slot group and see which scopes own it and which agents are granted it; select a workflow and see every binding that triggers it. These answer the questions composition diagrams exist for — "what does this module actually consist of?", "where else is this agent used?", "who can write this slot?" — at a glance instead of via four admin pages.
- **Click-through, never edit-in-place.** Every node deep-links to its existing editor (the agent pages, the workflow builder, the map editor, the module config form). The atlas navigates; the dedicated editors edit. This boundary is deliberate: a canvas that also edited everything would become a parallel mega-editor that drifts from the real forms.

Architecturally the atlas is **a pure projection with zero new schema**: every node and edge it draws is derived from data this design already maintains — the module registry, the binding pivots, slot definitions, workflow bindings, knowledge grants, and the published graph version — assembled by one read-only endpoint into a composition graph for the canvas. That is also why it stays honest automatically: it cannot disagree with reality, because it has no state of its own. Overlay toggles let the operational views (engagement heat, journey replay) render on the same picture, so "structure" and "what users are doing in it" are two lenses on one canvas rather than two tools.

One admin tool earns special mention because it is nearly free and disproportionately valuable: **journey dry-run**. Because the engine and guidance functions are pure, the admin can simulate a synthetic user against a _draft_ map — "given these completions, these slot values, and this date, what's available, what's locked and why, what would guidance rank first?" — before publishing. For a thirty-node map this is a nicety; for a three-hundred-node one it is the difference between publishing on faith and publishing on evidence. It also doubles as the natural home for testing temporal gates ("set the clock to deadline week") and policy effects.

## 6. Data-Slots — lightweight sketch

This section is deliberately a sketch: the shape the other two layers need to exist, drawn to fit Sunrise's conventions. The deeper design (capture-loop tuning, pacing, targeted/open mode mechanics) is deferred to a dedicated design pass that will analyse the prior slot implementation described in the concept notes. The sketch is shaped so that pass changes capture behaviour, not these schemas.

### 6.1 The same two-sided split

**Definitions** are authored configuration: what the system aims to learn. **Values** are runtime user data: what it has learned, insert-only and versioned. Definitions come from two sources that land in one table — globally-defined slots (seeded by the app) and module-owned slots (declared in each `ModuleDefinition`, upserted at registration, which is how "a module has its own data-slots" works without a second mechanism).

```prisma
// framework-data-slots.prisma (sketch)
model SlotDefinition {
  id          String  @id @default(cuid())
  slug        String  @unique
  group       String                    // thematic cluster
  description String  @db.Text          // what this slot means — also prompt material
  scope       String  @default("global")// global | module:<slug> | facilitation
  visibility  String  @default("open")  // open (user sees/edits) | hidden (system-only)
  mode        String  @default("targeted") // targeted | open (minted at runtime)
  dataType    String  @default("text")  // text | number | boolean | date | json —
                                        // gives gate conditions & analytics a typed form
  sensitivity String  @default("standard") // standard | sensitive | special_category —
                                        // drives masking-before-storage, retention policy,
                                        // capability-exposure defaults, consent requirements
  priorityWeight Int  @default(0)       // sequencing input, targeted mode
  isActive    Boolean @default(true)
  @@index([group, scope])
  @@map("framework_slot_definition")
}

model SlotValue {                       // INSERT-ONLY — never updated, never deleted (except erasure)
  id            String   @id @default(cuid())
  userId        String                  // satellite FK convention + erasure hook
  slotSlug      String                  // definition slug, or minted slug (open mode)
  version       Int                     // monotonic per (userId, slotSlug)
  value         String   @db.Text       // plain language — canonical for conversation
  valueJson     Json?                   // typed form per the definition's dataType —
                                        // canonical for gate conditions & analytics
                                        // ("mastery ≥ 7" reads this, never parses prose)
  confidence    Int                     // 1–10
  sourceType    String                  // direct | unprompted | emerged_naturally |
                                        // built_across_turns | inferred | user_confirmed | synthesised
  reasoningNote String   @db.Text       // one sentence: how this reading was made
  provenance    Json                    // { conversationId, messageRange?, moduleSlug?,
                                        //   nodeKey?, capturedAt, contextExcerptRef? }
  supersededAt  DateTime?               // set on the PREVIOUS head when a new version lands.
                                        // Deliberate denormalisation: the current picture of a
                                        // user is "every row WHERE supersededAt IS NULL" — one
                                        // indexed filter instead of a max(version) scan per slug.
                                        // Do not "simplify" this away; head reads are the hot path.
  capturedAt    DateTime @default(now())
  @@unique([userId, slotSlug, version])
  @@index([userId, capturedAt])         // "freshest slots" reads for guidance
  @@map("framework_slot_value")
}
```

The current picture of a user is the head version of each slug; the historical progression — the archive of the user's evolution — is the full version sequence, each row carrying its own provenance, as the concept notes require (provenance per version, traceable to the conversation, module, time, and means of capture). `provenance.conversationId` links to the existing `AiConversation` archive, so "follow the link to see the original exchange" is a join, not a new storage system. Confirmations and negations from the framework overview map cleanly: a confirmation is a new version with raised confidence and `sourceType: user_confirmed`; a contradiction is a new version whose reasoning note records the negation — and is a signal the guidance layer can treat as generative material.

Two fields above were adopted _now_ deliberately, because both are one-column decisions today and migration-plus-backfill exercises once values exist. **`dataType`/`valueJson`** exists because not every app's slots are prose: one app records mastery scores, another records dates and decision lists. The plain-language `value` remains canonical for conversation; the typed form is canonical for anything mechanical — gate conditions in the map ("requires mastery ≥ 7"), analytics, dashboards — so nothing ever string-parses prose to make a decision. **`sensitivity`** exists because some apps' slot values are special-category data under GDPR (health information; data about minors): the classification drives masking-before-storage in `fill_slot`, retention policy, stricter defaults for capability exposure, and any consent requirements — meaning the masking requirement from the concept notes hangs off declared classification rather than ad-hoc judgement at capture time.

### 6.2 Capture as capabilities

`get_state()` and `fill_slot(...)` are built-in framework capabilities (the prior implementation's tool surface, ported to `BaseCapability`), granted to agents through the normal pivot. `fill_slot` validates against the definition (or mints a slug in open mode), writes the next version, and is silent in the visible conversation — which is precisely how the existing tool loop already behaves. `processesPii = true` with an explicit `redactProvenance` override is mandatory here; the platform's registry will refuse the registration otherwise, which is the right kind of friction for this table. **Selective exposure** — "selected slots deliberately exposed to agents" — is per-binding configuration: the capability grant's `customConfig` carries which groups/scopes this agent may read and write, enforced inside the capability, so a module's agent sees its module's slots plus whatever global groups it's granted, and nothing else.

Open implementation questions from the concept notes stay open and are unaffected by this shape: in-loop capture vs a separate post-turn extraction pass (the capability surface serves both — an extraction pass is just another caller), version-every-change vs meaningful-changes-only (a write-policy inside `fill_slot`, not a schema question), and how aggressively to surface provenance to users. One new item joins that deep-design list: **open-mode slug curation.** Runtime-minted slugs are the part of the slot system most likely to sprawl (thousands of one-off slugs per user across a population). The deep design pass should include a curation/promotion story — minted slugs reviewed, merged, or promoted into proper `SlotDefinition` rows when a pattern recurs — so open-mode capture enriches the schema over time rather than diluting it.

### 6.3 Consumers

Modules write bespoke per-module data through their scoped slots and read what they're granted. Guidance reads recency-ordered heads (`@@index([userId, capturedAt])` exists for exactly this). The user-facing transparency surface (open slots: see, confirm, refine, delete) is a later consumer API; _delete_ in an insert-only table means a tombstone version (user-initiated retraction, itself part of the history) — with full erasure reserved for the GDPR hook.

### 6.4 GDPR

`SlotValue` (and `UserJourney`/`JourneyEvent` from §5) register erasure hooks and hand-written `ON DELETE CASCADE` FKs per the satellite convention. Slot values are among the most sensitive rows the platform will hold; field-level masking before storage belongs in the `fill_slot` capability and is driven by the definition's `sensitivity` classification (§6.1) — `special_category` values get the strictest treatment by default, and a fork handling health data or minors sets classifications as part of authoring its slot definitions, not as a retrofit.

### 6.5 Relationship to `AiUserMemory`

The existing `AiUserMemory` (flat key/value per user+agent) is the primitive this system supersedes for framework apps: slots are user-centric rather than agent-centric, versioned rather than overwritten, confidence- and provenance-bearing. Leave `AiUserMemory` untouched (non-framework agents still use it); document that framework-bound agents should be granted slot capabilities instead. Whether to eventually migrate or bridge is a low-stakes later decision.

## 7. Cross-cutting conventions

**One versioning principle, two mechanisms.** Restating the §3 principle verbatim: **authored definitions get immutable version chains; user data gets insert-only history; everything is timestamped; nothing is ever destructively overwritten.** For definitions (module config, the map, slot definitions, policies) the chain follows the workflow discipline: draft-on-edit, promote-on-publish, pinning where execution semantics demand it, non-destructive rollback. For user data (slot values, journey events) the mechanism is insert-only rows. This is the codebase's existing discipline (`VERSIONING.md`, workflow versioning, audit log) made universal.

**Free-form `String` status/type fields, never Prisma enums — a platform convention, kept deliberately.** Every status, kind, and type column in the sketches (`Module.status`, `UserNodeState.status`, `SlotDefinition.sensitivity`, edge/policy `kind`s, …) is a `String`, matching how the entire existing schema is written (`AiAgent.kind`, `AiKnowledgeDocument.status`). Rationale: Prisma enums make every new value a migration and complicate cross-fork merges, while the platform's established alternative — free-form `String` plus a raw-SQL CHECK constraint where integrity matters (the documented pattern in `orchestration-knowledge.prisma`) — keeps values evolvable and forks mergeable. An implementer who "improves" these into Prisma enums is diverging from the platform, not tidying it.

**One scoping vocabulary.** `moduleSlug` / `nodeKey` / `scope` appear with the same meaning on slots, agent bindings, the entries the framework writes into `CapabilityContext.scope`, workflow bindings, knowledge grants, and journey events. A small `lib/framework/shared/scope.ts` owns the types so the three domains can't drift — and so the vocabulary lives on the framework side of the boundary, never in core types.

**One agent-binding pattern, two boring tables.** Module seats and facilitation roles use the same pivot _shape_ (`agentId`, `role`, `config`) but deliberately stay two small parallel tables rather than one polymorphic `FrameworkAgentBinding`. With the relationship/cohort overlay (§8) on the horizon adding further association tables (`Cohort`, membership, journey links), the polymorphic-everything temptation is exactly the wrong trade in Prisma: several boring, identically-shaped tables age better than one clever generic one.

**Read paths go through an access check, from day one.** Every read of journey state or slot values goes through a single function — `canRead(viewer, subject, scope)` — rather than raw `where userId = session.user` queries scattered through handlers and capabilities. Today that function is one line: viewer must equal subject (plus admin support-tooling paths). The discipline costs nothing now, and it is the entire difference between the relational features in §8 being _a policy change inside one function_ versus _a sweep of every query in the codebase_. The same applies to journey analytics queries, which are written against a "subject scope" filter (one user today; a cohort tomorrow) from the start.

> **Compose with the Sunrise scope seam (issues #366/#367).** Two upstream fork-readiness proposals describe the _platform-tier_ version of this same authorization shape: #366 (org/tier axis — vendor-staff vs customer-admin vs member) and **#367 (intra-tenant ownership-scope — owner/team-scoped resource visibility)**. Both generalise Sunrise's `withAdminAuth` chokepoint into a single resolver: _one predicate, three orthogonal inputs_ — tier (#366), ownership (#367), org (the multi-tenancy playbook). Daybreak's `canRead(viewer, subject, scope)` is the framework-layer instance of the **same** predicate. **Binding design note:** `canRead` must be built to _compose with_ that resolver when it lands upstream — its `scope` argument carries the ownership/subject input #367 supplies — **not** as a private parallel scope-check that later has to be reconciled. #367's driver (different leaders each owning their own resources within one install) is the direct precursor to §8's per-subject and cohort-facilitator visibility, so even though Daybreak's first app (Lelanea) is single-user and does not need #366/#367 to ship, the access seam is shaped for them now. Neither issue blocks the fork: they merge down as inert seams when landed (see [[plan#Relationship to Sunrise]]).

**Standard platform wiring.** New admin APIs under `/api/v1/admin/framework/**` (inherit rate limiting via the policy table, guarded by `withAdminAuth()`), admin pages registered in the nav, every config mutation written to the audit log, public-surface additions (`registerModule`, `GraphStore`, the framework capabilities) recorded in `CHANGELOG.md` per the contract.

## 8. The relationship & cohort overlay — a designed seam, not a built feature

The framework's core is per-user by construction: one person, their slots, their journey. Two of the reference use cases (§1.1) break that quietly — one needs _groups_ with a facilitator role; another has _supporters_ whose journey is about someone else's situation. The right response is not to make the core multi-user (each student genuinely has their own progress; each supporter genuinely has their own journey), but to design a **relationship overlay** that future apps switch on. Nothing in this section is built in the initial phases; everything in it is _provided for_ by three deliberate provisions already woven into the sections above, so that turning it on later is additive.

**Cohorts** are a group construct over individual journeys: a group working the same map, group-level analytics, a facilitator role with visibility into member journeys. The cohort design pass must also cover **hierarchy** — a parent-cohort / group-of-groups construct (a community spanning many study groups, a collective-wide tier above per-book groups) — taken as a requirement into that pass rather than retrofitted after flat cohorts ship. When built, this is a `Cohort` table, a many-to-many membership table (a person can belong to several cohorts simultaneously — in two study groups, or supporting two different people), a nullable `cohortId` dimension on journeys/events for attribution, and cohort-aware aggregation in the analytics the admin already has. The two provisions that make this cheap are already in place: analytics queries written against a subject-scope filter, and the journey explorer designed knowing a group view will exist (§7).

**Relational links with consent** are the genuinely new machinery, deferred until an app makes them real: _user A may see defined parts of user B's journey or slots, with B's consent, possibly time-limited, always revocable_ (supporter↔supported, parent↔child, mentor↔mentee). When built, this is a `JourneyLink` table (viewer, subject, scope grant, consent record, expiry/revocation) enforced inside the one `canRead()` function — which is precisely why that function exists from day one. Consent records here are append-only like everything else user-facing in the framework.

**Overlapping contexts** are the subtle case the supporter scenario exposes, and the one provision that _did_ warrant a schema decision now: a person supporting two different people walks the same supporter map twice, in two contexts, at two paces. That is the `contextKey` on `UserJourney` (§5.2) — multiple journey instances of one graph for one user, distinguished by context (when links exist, the contextKey naturally identifies the linked subject). The commonality between the contexts is captured _for free_ by the architecture's existing split: **person-level slots are user-global** ("my own coping capacity", "my patterns", "my circumstances") and serve every context at once — that _is_ the overlap — while context-specific material (this particular situation, this journey's progress) lives per-journey, keyed by context. No merging machinery, no duplicated person-model: the slot/journey boundary already separates "who I am" from "what I'm currently walking through", and the overlay simply leans on it.

**A further seam, named but not designed: the shared subject.** Apps where the work centres on an entity _between_ users — a dispute being resolved, a co-parenting plan, a shared business — break a deeper assumption than cohorts or relational links: the subject of the data is no longer a person but a jointly-owned object, with per-party stances on shared facts, gates that span multiple parties' journeys, and erasure semantics over joint records. Such apps would require a **shared-subject overlay** (a Subject entity, party links, subject-scoped data with per-party stance, and a multi-party predicate family for the engine). It is deliberately not designed here: its hard questions need a committed app's product and legal input. What this document guarantees is only that the per-user assumption is _known and bounded_, not accidental — and that the existing seams already lean its way (`contextKey` can carry a subject context, `canRead(viewer, subject, scope)` already names a subject, slot `scope` is an open string, and the family-tagged condition format makes a future `parties:*` predicate family additive rather than a map-format migration).

**An authoring-side seam, named but not designed: content ownership for multi-expert deployments.** The deployment unit is the app, not the expert — "expert-led" is a genre description, and no Expert entity exists anywhere in this design — so a collective app (a publisher's library, several authors or podcasters under one roof) is simply an app whose expert is plural: one deployment, one user account, one user-global slot fingerprint, with per-expert knowledge bases, agent personas, maps, and modules, all expressible today as configuration. What such a deployment adds is **ownership scoping over authored entities**: expert A may edit expert A's maps, modules, agents, and knowledge and not expert B's — an ownership dimension on authored entities plus owner-scoped admin RBAC. This is content-authorship partitioning, _not_ user-data multi-tenancy, so it does not disturb the one-fork-per-app, single-tenant stance. It is deliberately not designed here: its shape (ownership granularity, shared-module governance, cross-expert slot-exposure defaults — D5's per-grant mechanism with deny-across-experts as the natural default, while the user-global person layer remains deliberately shared) should be settled with a real collective's governance needs. One placement consequence is binding now: a federation of _separate_ deployments sharing users, billing, and groups is the multi-tenant platform A1 rejects — so a creator collective is built as **one app from day one**; retro-federating separate forks is the expensive version of every multi-expert scenario.

The cost of all this today is three small things already paid for above: one column (`contextKey`), one function (`canRead`), one query discipline (subject-scope filters). The overlay itself gets a proper design pass when the third or fourth app makes it real.

## 9. Open items

This section holds only the items that **cannot responsibly be decided in this document**, each with the structural reason it stays open; every settled decision lives in its body section with its rationale, summarised in Appendix A. Nothing in Phases 1–6 blocks on any of them.

1. **Slot deep-design.** OPEN — _sequenced, not contentious._ Deferred to a dedicated design pass analysing the prior slot implementation (see Source documents); deciding capture mechanics before that analysis would discard its evidence. The §6 sketch is shaped so nothing in Modules/Facilitation changes when that pass lands. Agenda: capture-loop placement (in-loop vs post-turn extraction), write policy (every change vs meaningful changes), provenance surfacing, masking rules per sensitivity class, and open-mode slug curation/promotion (§6.2).
2. **Auto-approval risk taxonomy** (§5.5). OPEN — _empirical, not contentious._ The build decision is made: ship with `autoApprove: none` (every structure-change proposal requires human approval). What stays open is the classification of which change types are safe to auto-approve — an empirical question that needs a population of real proposals to answer; classifying hypothetically would be guessing. Revisit once the proposal pipeline has operating history.
3. **Consent model for relational links** (§8). OPEN — _externally gated._ The seam is designed and the build is deliberately trigger-based (Appendix B philosophy); the consent model itself (grant scopes, revocation, time-limits, capacity questions for health-adjacent and minors-involving apps) is legal-review territory and must be designed with the app that triggers it, not speculatively here.

## 10. Suggested build sequence

Phased so each stage ships something inspectable and nothing blocks on the still-open questions:

**Phase 1 — Module system core.** Schema + registry + seam, generic config form from Zod schemas, lifecycle/flag/window liveness, agent bindings. Proves the code/config split end-to-end with one trivial demo module.

**Phase 2 — Journey spine.** Graph schema + version chain, `GraphStore` (Postgres impl), deterministic engine with invariant validation, journey state + event log, erasure hooks. Pure backend; testable without any UI.

**Phase 3 — Slots sketch implementation.** Definitions + values tables, `fill_slot`/`get_state` capabilities, module-declared slot registration. Enough for guidance to have something fresh to read; deep design lands later without schema upheaval.

**Phase 4 — Guidance + chat integration.** The framework capability family, slot-informed ranking, context-builder injection, facilitation agent bindings. First moment the experience is _felt_ in a conversation.

**Phase 5 — Admin surfaces.** Map editor on the workflow-builder canvas (with region collapse/expand), module admin pages with stats from the event stream, journey explorer with overlays, the framework atlas (§5.6 — the zoomable composition view, a pure projection over the registries and binding tables), and the journey dry-run simulator (§5.6) — cheap to build once the engine exists, and it pays for itself on the first non-trivial map.

**Phase 6 — Governance & emergence.** Typed policy kinds (relevance gating, guard minimums, escalation pathways, auto-approval risk classes), proposal pipeline through the approval queue, pgvector overlay, evaluation wiring, proactive-guidance scheduling.

**Later, when an app needs it — the relationship & cohort overlay** (§8). Cohorts, membership, journey links with consent. Not scheduled; its three enabling provisions (contextKey, `canRead()`, subject-scope analytics) ship inside Phases 2 and 5 so this phase is additive when it comes.

## Appendix B — Repo placement: a separate fork of Sunrise

**Decision (Binding, rev 15 — supersedes rev 14's in-core-severable decision).** The framework lives in its **own fork/repo of Sunrise**, not in Sunrise core. Each in-genre app (Lelanea first) forks the framework repo; the framework repo itself tracks Sunrise upstream via the `building-on-sunrise` model (fix-in-place → classify → promote generic primitives upstream into Sunrise, carry genre-specific machinery in the framework repo). Clean framework↔Sunrise seams remain a **hard architectural requirement** — not now to keep the framework _deletable_, but to keep the framework repo cheap to re-base on a fast-moving Sunrise.

**Why this changed from rev 14.** Rev 14 chose in-core-severable, weighing the options purely on engineering merge-cost and assuming a framework-heavy, public-by-default portfolio. Three facts the original analysis lacked reverse the conclusion: (1) the framework is genre-specific and HCE's actual portfolio is framework-_light_ (Sunrise, the Hub, ConQuest, client work are out-of-genre — Lelanea is the lone in-genre app), so in-core would tax the majority of forks with strip-and-merge work; (2) the **HCE Hub** is being built to coordinate upstream→downstream Sunrise flow, which absorbs the "merge-relay" cost that was rev 14's single load-bearing argument against a separate repo; (3) welding a genre layer into Sunrise erodes its strategic value as a clean, general, open foundation. Licensing of the framework repo (open vs proprietary / open-core) is a separate, still-open choice that does **not** block the build. See [[placement-decision-memo]].

**The applicability assessment behind it.** The framework is domain-agnostic but **genre-specific**. It encodes four structural assumptions: an individual human at the centre; an evolving picture of that person (slots); progression worth mapping (facilitation); a product composed of places the user dwells in (modules). Apps matching that genre — expert-led coaching, guided learning, behaviour change, patient/supporter pathways, professional development — fit naturally, and the framework degrades gracefully when an app uses only a subset (an episodic app may barely use the map). Apps outside the genre — support bots, document/workflow automation, B2B analytics assistants, internal ops tools — gain nothing from it and pay real costs if it is welded into their platform: framework tables in every migration, irrelevant admin sections, GDPR-sensitive satellite tables on deployments that never needed them, public-surface/CHANGELOG churn, and cognitive pressure to force-fit ("make the support bot a one-node map") where a plain page and an agent would do. The framework is therefore an asset within the genre and a mild hindrance outside it; repo placement is a bet on the portfolio mix.

**Options considered** (decision reversed in rev 15 — see above).

| Option                                         | For                                                                                                                                                                                                                   | Against                                                                                                                                                                                                                     |
| ---------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **In core, welded**                            | One repo; one merge hop for apps; framework maintained beside the primitives it reuses (a guard or versioning change in orchestration adapts the framework in the same PR)                                            | Every non-framework app (the majority) carries framework weight; Sunrise-as-general-template gets less clean                                                                                                                |
| **In core, built severable** (rev 14's choice) | The in-core maintenance advantages; non-framework apps achieved by deletion via a strip script                                                                                                                        | Taxes the _majority_ (framework-light portfolio) with recurring strip-and-merge conflicts; the framework is public because core is, foreclosing the open-core option; dilutes Sunrise's generality                          |
| **Separate fork of Sunrise** (chosen, rev 15)  | Sunrise stays clean and general; framework weight only where wanted; licensing free (open or proprietary); the merge-relay is Hub-coordinated and modelled on `building-on-sunrise`; non-framework forks are pristine | A genuine upstream-merge cost on the framework repo, and seam-moving changes surface at merge time rather than in-PR — but concentrated in **one** HCE-controlled, Hub-managed repo and bounded by the enforced seams of X6 |

**What clean seams require (Binding).** The design is already most of the way there by following Sunrise conventions: own `framework-*.prisma` schema files, own `lib/framework/` subtree, own admin route group, registry-based wiring. In a separate-fork world this discipline stops being about _deletability_ and becomes about _merge survivability_ — the property that lets the framework repo re-base on a fast-moving (0.x) Sunrise without every upstream merge fighting. Two requirements are strict. First, **generic seams**: every framework→Sunrise touch point goes through a registration mechanism whose Sunrise side is framework-agnostic — the context-contributor registry on `buildContext()`, the generic `CapabilityContext.scope` map, admin-nav registration — all wired behind a single `initFramework()` call site, never inline edits to Sunrise files. The test is absolute: **upstream Sunrise contains zero framework vocabulary** — no field, type, or string that names a concept the framework invented. (A `moduleId` field on a Sunrise type fails this test; a generic `scope` map passes it. The failure mode being prevented: a generic primitive promoted upstream drags a framework concept into core, and the boundary silently rots.) Second, **boundary scope**: the framework's code is not only `lib/framework/` — it is every framework path (`lib/framework/`, `app/admin/framework/`, `app/api/v1/admin/framework/`, framework admin components). The boundary rule covers all of them, in both directions: no core or app-shell file imports from any framework path; framework paths import core only through its public seams. Enforcement: an ESLint boundary rule (the existing `lib/app/**` pattern) plus CI, shipped **in Phase 1** — enforcement cannot wait for a cleanup pass, because the temptation to shortcut through core peaks at Phase 4 (chat integration), and an unenforced boundary decays silently. The boundary is not wasted effort even setting merges aside — it is also what keeps the framework honest internally.

**Migration discipline (Binding).** Sunrise keeps one linear migration history applied in timestamp order. The framework repo inherits that history and adds its own DDL on top — which must always land in **its own migration folders, named with a `framework_` prefix, touching only `framework_*` tables — never mixed with Sunrise DDL in one folder**. Prisma generates a migration as a whole-schema diff, so a single mixed sitting produces interleaved SQL that fights every upstream merge (incoming Sunrise migrations tangle with framework DDL); keeping them prefixed and separate keeps `framework repo ← Sunrise` re-bases clean, and keeps any generic primitive promoted _upstream_ free of framework tables. Enforcement is a CI check (framework-prefixed migration folders may contain only `framework_*` DDL) plus review convention, since the ESLint boundary cannot see SQL. Erasure-hook registrations, seeds, and pgvector index DDL all live in `lib/framework/` or framework-prefixed migrations, under the same rule.

**The recurring cost, honestly: the upstream merge.** A separate framework repo does not escape merge work — it relocates it. On each `framework repo ← Sunrise` re-base the framework absorbs upstream changes to the Sunrise primitives it sits on; where an upstream change moves a seam the framework relies on, that surfaces at merge time rather than in the same PR (the one real advantage in-core would have kept). During Sunrise's `0.x` phase — whose own `VERSIONING.md` warns forks to expect real merge work between any two releases — this velocity is high. Three things keep it bounded: the cost lives in **one** HCE-controlled repo (not scattered across every non-framework fork, as in-core-severable would have had it); the enforced seams (X6) mean upstream changes mostly land without touching framework code; and the **HCE Hub** is being built to coordinate exactly this upstream→downstream flow. Each in-genre app then forks the framework repo and inherits Sunrise transitively through it.

**Promote upstream, don't accumulate.** Per `building-on-sunrise`: when the framework needs something generic that Sunrise lacks, fix-in-place then _classify_ — a generic primitive or seam is promoted **upstream into Sunrise** (keeping Sunrise richer and the framework thinner), while genre-specific machinery stays in the framework repo. Over time the framework repo feeds Sunrise rather than bloating it, and the in-genre apps below inherit the upstream improvement for free. This is the same philosophy the design applies to `GraphStore` (F8): commit to the boundary now; defer the second system until the workload exists.

**Continuous verification (Binding).** The boundary is proven by CI, not prose. On every build, CI enforces: (1) the ESLint import boundary across all framework paths in both directions; (2) zero framework vocabulary in Sunrise-side code (no framework-named field, type, or string on a core type); (3) migration hygiene (`framework_`-prefixed folders touch only `framework_*` tables). A green run means the framework repo is still cheap to re-base this week; a red run means a boundary leak entered this week — caught at commit time, not discovered at the next upstream merge. (Rev 14's `scripts/strip-framework.sh` is dropped: with the framework in its own repo there is nothing to strip.)

## Appendix C — Adjacent components: what a complete app needs beyond this framework

The framework plus the existing Sunrise platform do not constitute a complete consumer application. This appendix inventories the backend components a production app in the reference genre will additionally need. **None of them belongs inside the framework** — each is a sibling component with a narrow, definable interface to it — but several touch framework seams, and those interface points are flagged so the framework is built aware of them. Nothing here is scheduled by this document; each component arrives when an app needs it.

**C1 — Commercial layer.**

- **Billing & subscriptions** — payment provider integration, plans, trials, dunning, refunds, VAT/invoicing. Nothing in Sunrise touches money.
- **Entitlements** — distinct from feature flags: flags are operational rollout, entitlements are what a customer has paid for. _Framework interface:_ module liveness (§4.1) computes status × flag × window × journey gating; a paid tier is a fourth input, so the liveness check should accept an entitlement predicate when this component exists. **Earliest-relevant interface — touches Phase 1.**
- **Per-user cost budgets** — Sunrise estimates LLM cost operationally; consumer apps need per-user/per-plan quota enforcement (heavy chat use on a cheap plan is a real margin risk in this genre).
- **Promotions, referrals, gift access.**

**C2 — Identity & account lifecycle (beyond core auth).**

- **Consumer auth flows** — social login, magic links, MFA, self-service account deletion (erasure hooks exist; the user-facing flow does not).
- **Age assurance & parental consent** — hard requirement where minors are involved and likely for health-adjacent apps: verifiable consent flows, age gates, child-appropriate data handling.
- **Organisational accounts** — an institution buying seats: seat licensing, institutional SSO (SAML/OIDC), org-admin roles. _Framework interface:_ the commercial twin of the cohort seam (§8) — an org owns cohorts, but org/billing/SSO machinery is its own component.

**C3 — Communication & engagement channels.**

- **Channel adapters** — push (APNs/FCM), SMS, WhatsApp Business API, with template management and delivery tracking. _Framework interface:_ proactive guidance (§5.4) produces the nudge; this layer delivers it.
- **Communication preferences & consent** — transactional vs marketing, quiet hours, frequency capping, per-channel opt-in; legally required and behaviourally critical in this genre.
- **Deliverability operations** — bounce handling, suppression lists.

**C4 — Media & rich interaction.**

- **User media pipeline** — uploads (voice notes, photos, documents), storage, scanning, retention, CDN serving. The knowledge pipeline ingests _expert_ content; nothing handles _user_ content.
- **Voice** — STT/TTS for spoken check-ins or voice companions.
- **Real-time transport beyond chat streaming** — WebSocket/SSE presence and live group features; relevant the moment cohorts become interactive.

**C5 — Trust, safety & safeguarding operations.**

- **Abuse & fraud prevention** — signup fraud, bot detection, payment fraud. Guards protect agent output; nothing protects the platform from hostile users.
- **User-to-user content moderation** — required as soon as cohort members can see each other's content; distinct from agent guards.
- **Safeguarding case management** — escalation policies (§5.5) _detect and route_; a human then needs a case to work: assignment, status, notes, audit. For health-adjacent apps this is the difference between an alert and a duty-of-care process. _Framework interface:_ consumes escalation events.
- **Consent & policy-acceptance records** — versioned T&C/privacy acceptance; pairs with slot sensitivity machinery but is its own ledger.

**C6 — Product data & experimentation.**

- **Product analytics** — funnels, retention, activation. Journey events answer "how do users move through the map?"; product analytics answers "is the business working?" — different audience, different store. _Framework interface:_ consumes journey events.
- **Product-level A/B testing** — Sunrise's experiments evaluate agents; experimenting on pricing, onboarding flows, or copy is a separate capability.
- **Warehouse export / BI** — ETL of journey events, slots (sensitivity-filtered), and commercial data for external analysis.

**C7 — Platform engineering & internationalisation.**

- **Caching/session infrastructure** (Redis-tier) as load grows — rate limits, hot slot reads, session store.
- **High-volume job queue** — leased workflow executions may not suit million-message notification fan-out.
- **Public API / partner platform** — API keys, OAuth provider, customer-facing webhooks, developer portal, if experts or institutions integrate programmatically.
- **i18n / l10n** — translated content, multilingual agent policy, locale formatting. _Framework interface:_ temporal gate predicates (§5.1) evaluate `now` — against whose timezone? A deadline gate for a user abroad needs per-user timezone resolution, so the engine's `now` input should be timezone-resolved per journey. **Earliest-relevant interface — touches Phase 2.**
- **Mobile platform concerns** — device registry, minimum-version gating, offline sync if ever promised.

The pattern across all seven categories: each component has a narrow interface to the framework — entitlements feed module liveness, channels carry guidance outputs, org accounts own cohorts, case management consumes escalations, analytics consumes journey events — so none weakens the framework design. The two interfaces worth carrying from the start are **entitlements** (module liveness, Phase 1) and **timezone-resolved `now`** (temporal predicates, Phase 2); the rest arrive with the app that needs them.
